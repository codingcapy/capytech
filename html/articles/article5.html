Frameworks, Modules and Packages

Posted on: January 10, 2024

There's a lot of conflicting opinions among engineers about the use of frameworks, modules and libraries.

The main arguments of those on the opposing camp raise concerns over bloating the web and unnecessarily complicating things that don't need to be complicated. Although they have a point, there's a reason frameworks and modules like React, Next, Remix, Angular, Vue, Tailwind, Express and Bun have taken over the web development industry.

I recently returned to using vanilla HTML, CSS and JS after several months of using React, Node, Express, Tailwind and Next. And to be honest, it didn't feel too bad; I just had to work on html tags instead of jsx and tsx, and had to implement logic and axios api calls using javascript instead of inside my jsx and tsx. I did get bothered by having to use vanilla css and scss, so I just converted my styling into tailwind.

Development environment aside, I knew from the get-go that my vanilla project would have advantages over my MERN apps with regards to SEO and rendering speed.
